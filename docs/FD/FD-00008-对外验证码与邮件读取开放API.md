## 一、功能概述

### 1.1 设计目标

基于现有模块化工程，为系统补齐一套**对外开放的验证码与邮件读取 API**，重点服务于**本地化部署下的自动化注册流程**。

本次设计目标如下：

1. 在现有内部邮件读取能力之上，新增一层**API Key 鉴权的开放接口**。
2. 对外直接暴露**验证码提取**与**验证链接提取**能力，降低接入方二次开发成本。
3. 补齐**邮件列表 / 邮件详情 / RAW 内容**接口，支持人工排查与前端收件箱组件接入。
4. 提供**健康检查 / 能力查询 / 账号状态检查**接口，用于本地部署后的自检和联调。
5. 尽量复用现有 `controllers/emails.py`、`services/verification_extractor.py`、`services/graph.py`、`services/imap.py` 能力，避免重复实现。

### 1.2 核心约束

1. **不破坏现有内部 API**：`/api/emails/<email_addr>`、`/api/email/<email_addr>/<message_id>`、`/api/emails/<email_addr>/extract-verification` 保持原有行为与登录态约束。
2. **开放接口与后台登录态隔离**：外部接口不能依赖 Session/Cookie，只使用 `X-API-Key`。
3. **优先复用现有读取链路**：Graph → IMAP(New) → IMAP(Old) 的回退策略保持一致。
4. **不引入复杂多租户设计**：首版采用单实例单 `external_api_key` 模式。
5. **敏感配置不明文回显**：设置页中仅展示脱敏后的开放 API Key 状态。
6. **首版定位为受控开放接口**：面向本地化部署与单可信调用方，不以“可直接公网暴露的通用开放平台”作为本版交付目标。
7. **首版不承诺公网级防护与高并发能力**：`wait-message` 为同步轮询实现，IP 白名单、配额与开放平台级限流不在首版闭环内。

### 1.3 Out of Scope

- Webhook/回调式外发通知
- 多调用方 API Key 管理
- 附件下载与附件元数据开放
- 外部删除邮件、标记已读、移动邮件等写操作
- 通用开放平台能力（限流配额、租户体系、开发者门户）
- 公网模式下的完整安全治理（来源白名单、接口分级、调用方配额）

### 3.3 功能修改范围结论

本功能不是“新增几条 API 路由”即可交付，而是一次跨越**鉴权层、路由层、邮件读取服务层、系统自检层、设置页与文档层**的受控开放能力补齐。

从范围上看，本次需要明确区分三类边界：

1. **首版必须落地的范围**
   - 让外部调用方可以通过 `X-API-Key` 安全访问只读能力。
   - 让管理员可以在设置页配置、更新、清空 `external_api_key`。
   - 让开放接口复用现有 Graph → IMAP(New) → IMAP(Old) 读取链路，而不是单独重写读取逻辑。
   - 让系统对外暴露最小可联调能力：消息列表、最新邮件、详情、验证码、验证链接、基础健康信息。
   - 让文档口径与实现一致，明确“受控私有接入”而非“直接公网开放”。
2. **后续条件成立后再推进的范围**
   - 公网模式、IP 白名单、限流、高风险接口分级已在后续版本落地；真实上游探测、多 Key、范围授权仍属于后续演进范围。
   - `wait-message` 的异步探测与后台轮询解耦已落地首版，是否完全收敛同步模式仍不作为首版闭环前置条件。
3. **本轮明确不进入的范围**
   - 不改账号导入导出主流程。
   - 不改内部登录态 API URL 与主体行为。
   - 不增加邮件写操作、附件下载、Webhook 推送、多租户开发者平台能力。

因此，本 FD 对“修改范围”的最终定义是：

- **P0 首版交付范围**：受控私有接入闭环。
- **P1 扩展范围**：公网模式外围防护与接口分级。
- **P2 演进范围**：轮询解耦与权限模型升级。

#### 3.4.1 首版必改范围的功能边界

以上必改范围进一步落到功能边界时，应覆盖以下六块：

1. **开放访问入口**
   - 提供 `/api/external/*` 路由。
   - 保持与内部 API 路径隔离，避免复用登录态鉴权。
2. **开放鉴权与配置**
   - 新增 `external_api_key` 的存储、读取、脱敏展示。
   - 强制所有开放接口只接受 `X-API-Key`，不接受 query 参数中的 `api_key`。
3. **开放读取能力**
   - 支持消息列表、最新消息、详情、RAW、验证码、验证链接、等待消息。
   - 复用现有读取链路，不改变内部 Graph / IMAP 回退策略。
4. **最小系统自检能力**
   - 提供健康状态、能力列表、账号状态。
   - 允许调用方在接入前快速判断服务是否已配置与可调用。
5. **管理后台可维护性**
   - 设置页可查看是否已配置 API Key。
   - 支持手动更新或清空，不明文回显。
6. **文档与测试闭环**
   - 文档明确调用方式、返回结构、限制条件。
   - 测试至少覆盖成功、失败、未配置、参数错误、未命中等关键分支。

| 上游真实探测 | 当前 `health` / `account-status` 仅做轻量自检 | P1 |
| 多 API Key | 超出单可信调用方边界 | P2 |
| Key 级邮箱授权 | 超出单可信调用方边界 | P2 |
| `wait-message` 后台轮询解耦 | 当前首版允许同步轮询实现 | P2 |

### 3.5 条件后改范围

以下内容确实重要，但它们服务的是“公网暴露能力”或“长期平台化能力”，不应反向阻塞首版受控私有接入闭环。

| 范围 | 建议阶段 | 触发条件 | 说明 |
|------|----------|----------|------|
| 公网模式开关 | P1 | 需要通过公网或半开放网络提供 external API | 用于切换“私有接入”与“公网受限接入”两套规则 |
| IP 白名单 | P1 | 调用来源已知且希望只允许固定出口访问 | 适合作为公网前的第一层保护 |
| 请求限流 | P1 | 存在多个脚本、高频调用或外部扫描风险 | 保护单实例，避免挤占主线程 |
| 高风险接口分级 | P1 | 开始考虑公网暴露 | 公网模式默认禁用 `/wait-message` 与 `/raw` |
| 动态上游探测 | P1 | 需要更真实的“可读性”证明 | 让 `/health`、`/account-status` 不只停留在静态字段层面 |
| `wait-message` 解耦 | P2 | 并发等待请求增多，或需要稳定 SLA | 从同步轮询演进到后台轮询 / 缓存 / worker |
| 多 API Key | P2 | 一个实例对应多个调用方 | 为不同调用方建立独立身份 |
| Key 级邮箱授权范围 | P2 | 不同调用方只能读取部分邮箱 | 解决“一把 Key 读全实例”的权限过粗问题 |
| Key 启停、轮换、吊销 | P2 | 进入长期运营阶段 | 解决密钥泄漏与变更管理问题 |
| 调用方配额与审计增强 | P2 | 需要平台化运营 | 支持定位具体调用方与消耗行为 |

#### 3.5.1 为什么这些不放进首版闭环

原因不是这些能力不重要，而是它们对应的是另一层目标：

- 首版解决的是“**能否在受控环境稳定接入**”。
- P1 解决的是“**能否在半开放网络下相对安全地开放**”。
- P2 解决的是“**能否把这套能力做成长期可运营的外部平台接口**”。

如果在首版同时追求三层目标，改动面会从“开放接口能力补齐”扩大到“安全平台、调用方管理、异步架构重构”，风险和周期都会显著上升。

### 3.6 明确不改范围

为了防止需求蔓延，本功能以下范围明确不在本轮 FD 的实施边界内：

| 不改范围 | 说明 |
|----------|------|
| `outlook_web/routes/accounts.py`、`outlook_web/controllers/accounts.py` 主流程 | 不因为开放 API 反向重构账号管理模块 |
| 内部 API 路径与行为 | 保持 `/api/emails/*`、`/api/email/*` 等内部接口原有行为不变 |
| `telegram_push` 推送链路 | 与外部读取接口解耦，不纳入此次改造 |
| 附件下载、附件元数据开放 | 首版只聚焦验证码与邮件读取正文能力 |
| 邮件删除、标记已读、移动邮件 | 不开放写操作，避免权限与副作用扩大 |
| Webhook / 回调通知 | 不建设反向推送能力，仍由调用方主动拉取 |
| 多租户、开发者门户、套餐配额 | 不把当前项目升级为开放平台产品 |
| CSRF 主体逻辑 | 对外接口为 API Key 保护的 GET 只读接口，不引入额外 CSRF 设计 |

### 3.7 分阶段修改范围矩阵

为了让 FD、BUG、TDD 三份文档口径一致，功能修改范围按阶段收敛如下：

| 阶段 | 范围定位 | 必须修改 | 可暂缓修改 | 验收口径 |
|------|----------|----------|------------|----------|
| P0 | 受控私有接入闭环 | API Key 鉴权、开放路由、开放 controller、外部 service、设置页、基础自检、文档、测试 | 已完成 | 单实例、本地部署、单可信调用方可稳定接入 |
| P1 | 半开放部署增强 | 公网模式、IP 白名单、限流、高风险接口分级、动态上游探测 | 已完成前四项，动态上游探测待补 | 可以在受限公网环境下谨慎开放 |
| P2 | 平台化与可用性演进 | `wait-message` 解耦、多 Key、范围授权、吊销轮换、配额审计 | 已完成首版异步探测，其余待补 | 能支撑多调用方和更稳定的外部服务能力 |

进一步拆到实现层时，推荐按以下顺序推进：

1. 先完成 P0 文档口径与代码闭环，确保“受控私有接入”真实可用。
2. 再按部署场景决定是否进入 P1，而不是预先把所有公网治理一次性做完。
3. 当调用方数量、并发量、权限复杂度明显上升时，再启动 P2 的架构升级。

### 8.2 参数解析规范

所有开放接口统一使用轻量参数解析函数：

```python
def _parse_external_query_args() -> dict:
    return {
        "email": request.args.get("email", "").strip(),
        "folder": request.args.get("folder", "inbox").strip().lower(),
        "skip": int(request.args.get("skip", 0)),
        "top": min(int(request.args.get("top", 20)), 50),
```

### 8.3 统一响应包装

建议在 `services/external_api.py` 中提供：

```python
def ok(data: Any, message: str = "success") -> dict:
    return {"success": True, "code": "OK", "message": message, "data": data}


def fail(code: str, message: str, status: int, data: Any = None):
    return jsonify({"success": False, "code": code, "message": message, "data": data}), status
```

这样可以避免开放接口继续沿用内部 `{"success": False, "error": payload}` 的复杂结构。

### 8.4 `api_external_get_messages`

职责：

1. 校验 `email`、`folder`、分页参数
2. 查询账号是否存在
3. 调用 `external_api.list_messages_for_external()`
4. 返回统一结构
5. 记录审计日志

### 8.5 `api_external_get_latest_message`

职责：

- 基于 `messages` 列表结果取第一条，减少调用方二次处理
- 当无命中邮件时返回 `MAIL_NOT_FOUND`

### 8.6 `api_external_get_message_detail`

职责：

- 依赖 query 参数中的 `email`
- 读取指定邮件详情
- 返回 text/html/raw 的聚合结构

说明：

由于内部现有详情接口路径为 `/api/email/<email_addr>/<message_id>`，而开放接口拟采用 `/api/external/messages/<message_id>?email=...`，因此开放接口需要通过 query 参数重新传入 `email` 以定位账号。

### 8.7 `api_external_get_message_raw`

职责：

- 调用详情接口能力
- 仅返回 `raw_content`
- 用于前端调试或脚本兜底

### 8.8 `api_external_get_verification_code`

职责：

1. 获取符合条件的最新邮件
2. 拉取完整详情
3. 调用增强后的 `extract_verification_info_with_options()`
4. 若 `verification_code` 不存在，返回 `VERIFICATION_CODE_NOT_FOUND`

### 8.9 `api_external_get_verification_link`

职责与验证码类似，只是关注 `verification_link` 字段。

### 8.10 `api_external_wait_message`

#### 8.10.1 行为定义

- 在单次请求生命周期内循环轮询
- 到达超时时间仍未命中，则返回 `MAIL_NOT_FOUND`
- 默认参数：
  - `timeout_seconds=30`
  - `poll_interval=5`
- 最大超时时间限制为 120 秒，避免阻塞过久

#### 8.10.2 伪码

```python
start = time.time()
while time.time() - start < timeout_seconds:
    result = get_latest_message_for_external(...)
    if result["matched"]:
        return ok(result)
    time.sleep(poll_interval)
return fail("MAIL_NOT_FOUND", "等待超时，未检测到匹配邮件", 404)
```

```json
  "data": {
    "status": "ok",
    "service": "outlook-email-plus",
    "server_time_utc": "2026-03-08T13:00:00Z",
    "database": "ok"
  }
}
```

与现有 `healthz()`、`api_system_health()` 的区别：

| 接口 | 鉴权 | 目标 |
|------|------|------|
| `/healthz` | 无 | 容器探活 |
| `/api/system/health` | 登录态 | 管理员诊断 |
| `/api/external/health` | API Key | 外部接入自检 |

### 9.2 `api_external_capabilities`

返回当前实例支持能力列表，首版可写死并与代码实现同步：

```python
FEATURES = [
    "message_list",
    "message_detail",
    "raw_content",
    "verification_code",
    "verification_link",
    "wait_message",
]
```

### 9.3 `api_external_account_status`

功能：检查指定邮箱是否存在、是否可以读取。

最简实现：

1. 校验 `email`
2. 查询 `accounts_repo.get_account_by_email()`
3. 返回账号是否存在及基本状态
4. **不在该接口中真正拉信**，避免副作用与慢响应

建议字段：

```json
{
  "exists": true,
  "account_type": "outlook",
  "provider": "outlook",
  "group_id": 1,
  "last_refresh_at": "2026-03-08 20:00:00",
  "preferred_method": "graph"
}
```

2. `saveSettings()`：将 `external_api_key` 提交给 `/api/settings`
3. `generateExternalApiKey()`：前端生成随机值（长度建议 32-48）

建议随机生成逻辑：

```javascript
function generateExternalApiKey() {
  const bytes = new Uint8Array(24);
  crypto.getRandomValues(bytes);
  return Array.from(bytes, b => b.toString(16).padStart(2, '0')).join('');
}
```

## 十五、风险与决策记录

### 15.1 风险

| 风险 | 影响 | 缓解方式 |
|------|------|----------|
| `controllers/emails.py` 继续膨胀 | 可维护性下降 | 新增 `services/external_api.py` 承接逻辑 |
| API Key 明文泄露 | 安全风险 | 落库加密、设置页脱敏、日志不打印明文 |
| `wait-message` 阻塞请求线程 | 并发能力下降 | 限制最大 120 秒，默认 30 秒 |
| 外部接口与内部响应结构不一致 | 接入混乱 | 开放接口统一使用 `success/code/message/data` |
| 现有验证码提取误判 | 自动化稳定性不足 | 引入 `code_regex`、`code_length`、`code_source` 可配置项 |
| 单 `external_api_key` 读取全实例邮箱 | 权限边界过粗 | 首版仅按单可信调用方交付，后续再引入多 Key / 范围授权 |
| 缺少公网防护（白名单 / 限流 / 配额） | 公网暴露风险高 | 当前不以公网开放为首版目标，后续通过“公网模式”分阶段补齐 |
| 自检接口偏静态 | 可能“自检成功但实际不可读” | 后续补充上游探测与动态健康指标 |

### 15.2 关键设计决策

1. **开放接口与内部接口分离，不直接复用原 URL**。
2. **根项目的开放 API Key 使用加密存储**，不沿用示例项目的明文方案。
3. **开放接口优先返回统一简洁结构**，不直接透出内部 `build_error_payload()` 的嵌套结构。
4. **`account-status` 为轻量检查，不做真实拉信**，避免把自检接口做成慢接口。
5. **`wait-message` 当前同时保留同步轮询与异步探测两种模式**，其中 `mode=async` 已通过后台探测任务落地。
6. **首版开放 API 的正式定位为“受控私有接入”**，文档与实现均不以“直接公网暴露”作为验收标准。
7. **若未来支持公网暴露**，优先补充公网模式、来源白名单、接口分级与动态探测，而不是继续扩大开放接口集合。

## 十七、结论

本 FD 的核心思路是：

- **不重做邮件读取内核**，而是在现有模块化工程上补一层开放 API；
- **把“验证码/链接提取”升级为一等能力**，而不是让调用方自己解析邮件正文；
- **同时补齐人工排查与部署自检接口**，让本地化部署后的接入、排障、联调都可闭环；
- **通过新服务层 `external_api.py` 控制复杂度**，避免继续把逻辑堆进 controller。
