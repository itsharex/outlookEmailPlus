## 一、架构概述

### 1.1 目标

本架构文档用于说明“**对外开放 API 安全接入模块**”在当前项目中的定位、模块拆分、分层边界和后续演进方向。

这个模块的目标不是单纯多加几条 `/api/external/*` 路由，而是建立一套：

1. 能复用现有邮件读取能力的对外只读接口层
2. 能与后台登录态彻底隔离的 API Key 接入层
3. 能持续演进到公网安全治理和多调用方模型的架构基础

### 1.2 当前架构定位

当前架构定位明确为：

- **P0：受控私有接入**
- 面向**单实例**
- 面向**单可信调用方**
- 面向**本地部署 / 内网 / 受控访问环境**

当前不把该模块定义为“可直接公网暴露的开放平台模块”。

### 1.3 架构原则

1. **复用优先**：优先复用现有 Graph / IMAP / 提取器 / 设置 / 审计能力
2. **边界清晰**：外部 API 与内部登录态 API 路径、鉴权、响应口径隔离
3. **最小暴露**：首版只开放只读接口，不开放写操作
4. **安全收敛**：首版只接受 `X-API-Key`，不接受 query 参数传 Key
5. **渐进演进**：P0 先闭环，P1 再补公网防护，P2 再做平台化升级

## 二、新模块定义

### 2.1 模块名称

建议把本次新增能力统一视为一个新模块：

**External Open API Security Module**

中文可定义为：

**对外开放 API 安全接入模块**

### 2.2 模块职责

该模块负责：

1. 为外部调用方提供统一的 `/api/external/*` 访问入口
2. 通过 `X-API-Key` 建立独立于后台登录态的接入边界
3. 编排现有邮件读取链路，输出统一对外结构
4. 提供验证码、验证链接、详情、RAW、自检等能力
5. 记录开放接口调用审计
6. 为后续公网治理、权限模型、多调用方能力预留演进点

### 2.3 模块不负责

该模块当前不负责：

1. 多租户调用方管理
2. 公网模式完整治理
3. 应用层限流和配额平台
4. 异步消息中心或后台轮询平台
5. 邮件写操作、附件下载、Webhook 推送

## 三、总体架构分层

### 3.1 分层视图

建议将该模块拆成七层：

```text
外部调用方
  ↓
API 路由入口层
  ↓
外部鉴权与安全控制层
  ↓
Controller 协议适配层
  ↓
External API 编排服务层
  ↓
读取适配与提取能力层
  ↓
配置 / 仓储 / 审计 / 数据持久层
```

### 3.2 各层作用

| 层级 | 当前对应文件 | 作用 |
|------|--------------|------|
| API 路由入口层 | `outlook_web/routes/emails.py` `outlook_web/routes/system.py` | 对外暴露 `/api/external/*` 路径 |
| 外部鉴权与安全控制层 | `outlook_web/security/auth.py` `outlook_web/security/external_api_guard.py` | 校验 `X-API-Key`，并承接公网模式控制 |
| Controller 协议适配层 | `outlook_web/controllers/emails.py` `outlook_web/controllers/system.py` | 参数解析、错误映射、审计触发、HTTP 返回 |
| External API 编排服务层 | `outlook_web/services/external_api.py` | 统一业务编排、筛选、等待、错误码抽象 |
| 读取适配与提取能力层 | `outlook_web/services/graph.py` `outlook_web/services/imap.py` `outlook_web/services/imap_generic.py` `outlook_web/services/verification_extractor.py` | 实际读信和验证码/链接提取 |
| 配置与仓储层 | `outlook_web/repositories/settings.py` `outlook_web/repositories/accounts.py` `outlook_web/repositories/groups.py` | 读取 API Key、邮箱账号、分组代理配置 |
| 审计与持久层 | `outlook_web/audit.py` `outlook_web/db.py` | 审计日志与设置持久化 |

## 四、模块拆分结果

### 4.1 这个新模块需要哪几个部分

从架构上，这个新模块至少需要以下 8 个部分：

| 编号 | 部分 | 是否已具备 | 说明 |
|------|------|------------|------|
| M1 | 外部路由入口 | 已具备 | `/api/external/*` 路由 |
| M2 | API Key 鉴权层 | 已具备 | `api_key_required()` |
| M3 | 外部协议适配层 | 已具备 | 外部 controller 与统一响应 |
| M4 | 外部编排服务层 | 已具备 | `services/external_api.py` |
| M5 | 读取适配层 | 已具备 | Graph / IMAP / Generic IMAP |
| M6 | 提取能力层 | 已具备 | `verification_extractor` |
| M7 | 配置与运营层 | 已具备 | `external_api_key` 设置、设置页入口 |
| M8 | 审计与可观测层 | 已具备 | `audit_logs(resource_type='external_api')` |

但如果要继续演进，还需要补以下 4 个部分：

| 编号 | 部分 | 当前状态 | 目标阶段 |
|------|------|----------|----------|
| M9 | 公网安全控制层 | 已独立实现（`external_api_guard.py`） | P1 |
| M10 | 动态探测与健康增强层 | 未独立实现 | P1 |
| M11 | 异步等待 / 后台轮询层 | 已实现首版（probe + scheduler） | P2 |
| M12 | 多调用方权限模型层 | 未独立实现 | P2 |

### 4.2 当前建议的模块边界

为了避免后续继续把逻辑堆进 controller，建议以如下边界理解该模块：

#### M1 路由入口层

职责：

- 注册外部路径
- 保持与内部接口路径隔离
- 不承载业务逻辑

对应文件：

- `outlook_web/routes/emails.py`
- `outlook_web/routes/system.py`

#### M2 鉴权与安全控制层

职责：

- 校验 `X-API-Key`
- 阻断未配置 / 无效请求
- 后续承接公网模式、白名单、限流、接口分级

当前文件：

- `outlook_web/security/auth.py`

后续建议：

- 保留 `api_key_required()`
- P1 新增 `enforce_external_api_public_controls()` 一类扩展入口

#### M3 Controller 协议适配层

职责：

- 解析 query / path 参数
- 调用 service
- 把领域异常映射成统一 HTTP 响应
- 记录审计日志

当前文件：

- `outlook_web/controllers/emails.py`
- `outlook_web/controllers/system.py`

边界要求：

- 不直接写 Graph / IMAP 读取细节
- 不在 controller 中重复写提取算法

#### M4 External API 编排服务层

职责：

- 统一封装对外业务能力
- 聚合账号读取、过滤、详情、等待逻辑
- 统一错误模型与返回结构

当前文件：

- `outlook_web/services/external_api.py`

这是当前新模块中最核心的编排层。

#### M5 读取适配层

职责：

- 对接 Graph API
- 对接 Outlook IMAP
- 对接 Generic IMAP
- 保持 Graph → IMAP(New) → IMAP(Old) 回退顺序

当前文件：

- `outlook_web/services/graph.py`
- `outlook_web/services/imap.py`
- `outlook_web/services/imap_generic.py`

#### M6 提取能力层

职责：

- 统一从邮件正文中提取验证码、验证链接
- 支持自定义正则、长度范围、提取来源

当前文件：

- `outlook_web/services/verification_extractor.py`

#### M7 配置与运营层

职责：

- 持久化 `external_api_key`
- 向设置页提供脱敏展示
- 提供管理员可维护入口

当前文件：

- `outlook_web/controllers/settings.py`
- `outlook_web/repositories/settings.py`
- `templates/index.html`
- `static/js/main.js`
- `outlook_web/db.py`

#### M8 审计与可观测层

职责：

- 记录外部接口调用审计
- 输出 endpoint、email、method、error_code 等核心字段
- 为后续调用方追踪做基础

当前文件：

- `outlook_web/audit.py`
- `outlook_web/services/external_api.py`

## 七、安全边界设计

### 7.1 当前安全边界

P0 阶段当前架构已经建立的安全边界：

1. 外部接口与内部登录态接口路径隔离
2. 外部接口统一只接受 `X-API-Key`
3. `external_api_key` 不在设置页明文回显
4. 外部接口只开放只读能力
5. 调用行为进入审计日志

### 7.2 当前架构仍未补齐的安全部分

以下部分不属于 P0 闭环，但在架构上必须标识为“预留位”：

| 能力 | 当前状态 | 风险 |
|------|----------|------|
| 公网模式开关 | 已实现 | 可区分私网与公网策略 |
| IP 白名单 | 已实现 | 可限制来源访问 |
| 应用层限流 | 已实现 | 可做分钟级限流 |
| 高风险接口禁用开关 | 已实现 | `/raw` 与 `wait-message` 已具备公网降级能力 |
| 多 API Key 隔离 | 未实现 | 一把 Key 读取全实例邮箱 |

### 7.3 后续安全控制层设计

P1 建议新增一个“**公网安全控制子模块**”，统一放在 security / middleware 层处理：

职责：

1. 判断当前是否 `public_mode`
2. 获取真实客户端 IP
3. 做 IP 白名单校验
4. 做接口分级控制
5. 做单 IP / 单 Key 限流

建议逻辑顺序：

```text
api_key_required
  ↓
public_mode guard
  ↓
IP whitelist guard
  ↓
feature restriction guard
  ↓
rate limit guard
  ↓
controller
```

## 八、为什么这个模块要这样拆

### 8.1 不把所有逻辑放进 Controller

如果把参数解析、账号查询、读取回退、错误码映射、提取逻辑、轮询逻辑都塞进 controller，会导致：

1. 文件膨胀
2. 测试困难
3. 后续 P1/P2 扩展时难以插入安全控制
4. 同类能力无法复用

因此当前必须保留 `external_api.py` 作为独立编排层。

### 8.2 不直接新造一套邮件读取模块

因为现有项目已经有：

- Graph 读取
- Outlook IMAP 回退
- Generic IMAP
- 验证码提取器
- 设置仓储

所以架构上正确做法是“**编排复用**”，不是“**平行重造**”。

### 8.3 为什么 `wait-message` 不应该长期放在同步请求线程

因为它当前是：

- 同步轮询
- 请求线程阻塞
- 对单 worker 部署不友好

所以它在架构上只能算 P0 临时实现，不应该被当成长期公网能力模型。

## 九、分阶段架构演进

### 9.1 P0 架构

目标：

- 先把受控私有接入闭环做完整

模块状态：

- M1-M8 完成
- M9-M12 不进入当前实现

### 9.2 P1 架构

目标：

- 在不改业务接口路径的前提下补公网防护

新增模块：

1. 公网模式控制子模块
2. 白名单与限流子模块
3. 动态健康探测子模块

主要变更点：

- `security/auth.py` 或相邻安全层增加扩展 guard
- `controllers/system.py` 返回更多探测结果字段
- `/raw` 与 `/wait-message` 增加分级控制

### 9.3 P2 架构

目标：

- 支撑多个调用方和更高稳定性要求

新增模块：

1. 后台轮询 / 事件缓存子模块
2. 多 API Key 权限模型子模块
3. 调用方配额与吊销管理子模块

主要变更点：

- `wait-message` 从同步等待转向后台轮询或缓存查询
- 配置模型从单 `external_api_key` 升级为多 Key 结构
- 审计日志从“记录请求”升级为“记录调用方行为”

## 十一、实施结论

### 11.1 当前这个新模块已经做好的部分

目前从架构角度看，已经做好的部分是：

1. 路由入口已建好
2. API Key 鉴权已建好
3. 外部 controller 已建好
4. 外部编排 service 已建好
5. 读取适配与提取能力已接好
6. 设置页和配置链路已接好
7. 审计与文档链路已接好

基于当前实现状态，可以判断 **P0 架构已经基本成型**。

### 11.2 现在还需要哪几个部分

如果下一阶段要继续做，建议优先顺序如下：

1. **P1 第一优先级**
   - 公网模式控制
   - IP 白名单
   - 限流
   - 高风险接口分级
2. **P1 第二优先级**
   - 动态健康探测
   - `health` / `account-status` 增强
3. **P2 第一优先级**
   - `wait-message` 解耦
   - 后台轮询 / 缓存 / worker
4. **P2 第二优先级**
   - 多 API Key
   - Key 级邮箱范围授权
   - 吊销、轮换、配额

### 11.3 最终建议

当前不要再把重点放在“新增更多外部业务接口”上，而应该按下面两条主线继续演进：

1. **鉴权收敛主线**
   - 从单 `X-API-Key` 逐步演进到公网模式、白名单、限流、多 Key、范围授权
2. **轮询解耦主线**
   - 从同步 `wait-message` 逐步演进到后台轮询、缓存、异步等待模型

这两条主线才是该模块后续架构升级的核心。
