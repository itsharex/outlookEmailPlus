# BUG-00017 验证码提取能力准确率与一致性综合问题

> 关联需求: `PRD-00008` / `FD-00008` / `TDD-00008`
> 记录时间: 2026-03-12
> 影响范围: UI 列表页提取、UI 详情页提取、`/api/external/verification-code`、`/api/external/verification-link`
> 关联 BUG: `BUG-00015` / `BUG-00016`
> 当前状态: 汇总分析版，作为后续整改与排期的统一入口

---

## 一、问题概述

| 编号 | 标题 | 严重程度 | 类型 |
|------|------|---------|------|
| **VX-1** | 当前系统“提取验证码/验证链接”能力同时存在准确率缺陷与链路不一致问题，已经影响自动化接入可信度与产品语义一致性 | 🟠 高 | 业务准确率 / 架构一致性 |

## 二、统一结论

当前问题不能再拆成彼此独立的两个小问题来看：

1. `BUG-00015` 暴露的是对外 API 的直接业务缺陷  
   即：营销邮件中的普通数字和普通链接会被误提取成验证码或验证链接。

2. `BUG-00016` 暴露的是更深层的结构性问题  
   即：系统内部至少存在三套提取链路，缺少统一提取核心、统一输入模型、统一成功判定标准。

这两个问题不是并列关系，而是“症状 + 根因”的关系：

- `BUG-00015` 是当前最直接、最影响外部调用方的症状
- `BUG-00016` 是导致这类问题反复出现、难以单点修完的结构性根因

因此，后续讨论、排期、修复、测试，都应优先以本汇总文档为主，而不是继续把两个问题割裂处理。

## 三、问题表现

### 3.1 对外 API 的直接异常表现

在宽松筛选条件下，例如：

```http
GET /api/external/verification-code?email=...&since_minutes=10080
GET /api/external/verification-link?email=...&since_minutes=10080
```

系统可能命中一封并非验证码邮件的营销邮件，并返回：

- 一个普通数字作为 `verification_code`
- 一条普通跳转链接作为 `verification_link`

而且接口会以 `200 OK` 的形式返回成功，只附带 `confidence=low`。

### 3.2 UI 与对外 API 结果不一致

当前“提取验证码”至少存在三条用户可见链路：

1. UI 列表页/分组页按钮
2. UI 邮件详情页按钮
3. 对外开放 API

这三条链路不共享同一提取器，也不共享同一输入模型，因此同一封邮件在不同入口上得到的结果可能不同。

## 四、现状拆解

### 4.1 UI 列表页 / 分组页

列表页“提取验证码”调用：

```text
GET /api/emails/{email}/extract-verification
```

最终走的是旧提取链路：

```python
extract_verification_info(email_obj)
```

特点：

- 走后端内部接口
- 使用旧版提取器
- 规则相对简单
- 没有对外 API 的参数化能力

### 4.2 UI 详情页

详情页按钮直接在前端读取当前邮件正文，再做浏览器端正则提取。

特点：

- 不走后端统一提取服务
- 直接从 DOM / iframe 取正文文本
- 自己抓 `4-8` 位数字或字母组合
- 自己抓第一条可见 URL

### 4.3 对外 API

对外接口：

```text
GET /api/external/verification-code
GET /api/external/verification-link
```

最终走扩展提取链路：

```python
extract_verification_info_with_options(...)
```

特点：

- 支持 `code_regex`
- 支持 `code_length`
- 支持 `code_source`
- 支持链接关键词优先级
- 返回 `confidence`

## 五、根因分析

### 5.1 直接原因

对外 API 当前采用了“最新邮件 + 宽松 fallback + low confidence 仍成功”的组合策略。

具体表现为：

1. 邮件选择偏宽松  
   系统不是在候选邮件中挑最像验证码邮件的一封，而是：
   - 先按 `from_contains` / `subject_contains` / `since_minutes` 做基础过滤
   - 直接取“最新一封”
   - 再对这封邮件做提取

2. 验证码提取偏宽松  
   当关键词强匹配失败时，会全局扫描正文中的 `4-8` 位数字候选，并返回第一个满足基本过滤条件的值。

3. 链接提取偏宽松  
   当没有命中 `verify` / `confirm` / `activate` / `validation` 等关键词时，会直接回退到正文中的第一条有效链接。

4. `confidence=low` 不阻断成功  
   目前 controller 只要看到 `verification_code` 或 `verification_link` 非空，就按成功返回，不会因为 `confidence=low` 转成失败或待确认状态。

### 5.2 设计原因

当前提取能力本质上更接近“候选信息提取器”，而不是“严格验证码识别器”。

但对外接口名称、UI 按钮名称以及接入方预期，都会自然把它理解成：

- 可以直接自动化消费的验证码接口
- 结果应当稳定、可信、可复现

这就导致“实现语义”和“产品语义”发生了偏差。

### 5.3 结构原因

系统内部没有统一的提取契约，最终形成了三套并存实现：

1. 旧版内部提取器
2. 外部 API 参数化提取器
3. 前端本地正则提取

缺失的关键统一契约包括：

1. 输入契约  
   到底基于正文、HTML、subject、preview 中哪些内容提取。

2. 规则契约  
   到底什么叫验证码、什么叫验证链接，是否允许 fallback。

3. 成功契约  
   是只要拿到候选值就成功，还是只有高置信度结果才成功。

4. 一致性契约  
   UI、内部 API、对外 API 是否必须在同一封邮件上得出一致结论。

### 5.4 技术根因补充

三条链路不仅规则不同，连输入数据都不同：

1. 旧提取器  
   通常按 `body -> body_html转文本 -> body_preview -> subject` 回退。

2. 外部 API 提取器  
   默认把 `subject + content + html` 拼接后再做参数化提取。

3. UI 详情页前端提取  
   直接读取浏览器可见文本，再用前端正则抓值和抓链接。

输入模型不同，结果不一致是必然的，不是偶发。

### 5.5 测试根因

当前测试更多验证“每条链路各自可用”，但没有约束“跨链路结果应一致”。

缺失的关键测试包括：

1. 同一封标准验证码邮件，UI / 内部 API / 外部 API 结果一致
2. 同一封营销邮件，三个入口都不能把普通数字当成可信验证码
3. `confidence=low` 是否应影响最终接口成功语义

## 六、影响评估

### 6.1 对自动化接入的影响

- 自动注册流程可能误用营销邮件数字
- 自动点击流程可能跳到错误链接
- 接入方会遇到“接口成功但业务失败”的高排查成本问题

### 6.2 对产品语义的影响

- 用户看到“提取验证码”会默认认为不同入口结果一致
- 当前实现破坏了这个预期
- 会削弱 UI 与 external API 的可信度

### 6.3 对维护与测试的影响

- 后续每修一个提取规则，都要同时判断是否改三套实现
- 任何单点修复都可能造成另一条链路继续分叉
- 没有统一契约时，测试也很难真正兜住回归风险

---

## 七、问题归类

建议统一按“三层问题”理解：

1. 第一层：用户可见症状  
   外部 API 会误提取营销邮件数字/链接。

2. 第二层：直接实现缺陷  
   “最新邮件 + 宽松 fallback + low confidence 仍成功”的策略导致错误被成功返回。

3. 第三层：结构性根因  
   系统内部缺少统一提取核心和统一契约，导致 UI 与 external API 长期分叉。

---

## 八、处理建议

### 8.1 当前阶段处理口径

当前不建议继续把这类问题拆成互不相干的小修小补，而应按“统一提取能力”方向处理。

现阶段可以先明确以下口径：

1. `BUG-00015` 属于直接业务缺陷，优先级高
2. `BUG-00016` 属于结构性问题，不必推翻当前 PRD/TODO 阶段结论，但必须进入后续整改计划
3. 在统一提取核心前，不应默认宣称 UI 和 external API 语义一致

### 8.2 建议整改顺序

建议按最小风险顺序推进：

1. 先修 external API 的直接误判问题  
   目标：降低营销邮件误命中、让 `low confidence` 不再直接伪装成成功。

2. 再统一提取核心  
   目标：让 UI 详情页、UI 内部接口、external API 收敛到同一个后端 service。

3. 最后补一致性测试  
   目标：把“同一邮件跨链路一致”变成自动化约束。

## 九、需要修改的范围

本节用于明确“后续真正要改哪里”，避免把问题继续理解成单点正则修补。

### 9.1 第一层：必须修改的核心实现范围

这部分属于真正决定问题是否能止血的核心范围。

#### A. 外部 API 编排层

**涉及文件：**

- `outlook_web/services/external_api.py`
- `outlook_web/controllers/emails.py`

**需要调整的内容：**

1. `get_verification_result()` 的邮件选择策略  
   当前是“取最新一封再提取”，后续至少要改成：
   - 先拿候选邮件集
   - 再基于验证码/验证链接语义挑选最合适的一封
   - 或在 strict 模式下对低质量候选直接判定为未命中

2. controller 的成功判定语义  
   当前只要 `verification_code` 或 `verification_link` 非空就返回成功。  
   后续要明确：
   - `confidence=low` 是否允许继续 `200`
   - 是否需要新增 `LOW_CONFIDENCE` / `VERIFICATION_NOT_CONFIDENT` 一类业务错误
   - 是否支持 strict / loose 两种模式

3. `verification-code` 与 `verification-link` 的结果契约  
   当前两者都复用同一个提取流程，但业务语义不同。  
   后续需要明确：
   - 验证码接口的最低可信门槛
   - 验证链接接口的最低可信门槛
   - 两者是否共享一套 confidence 语义，还是拆成 `code_confidence` / `link_confidence`

#### B. 提取器核心层

**涉及文件：**

- `outlook_web/services/verification_extractor.py`

**需要调整的内容：**

1. 统一旧提取器与参数化提取器的职责边界  
   当前同时存在：
   - `extract_verification_info()`
   - `extract_verification_info_with_options()`

   后续要决定是：
   - 保留一个统一核心，另一个变薄包装
   - 还是完全废弃旧入口，统一收敛到增强版核心

2. 明确输入模型  
   需要统一：
   - 是否默认包含 `subject`
   - 是否默认包含 `body_html`
   - `body_preview` 的回退边界
   - UI / internal API / external API 的输入组装方式

3. 明确置信度模型  
   当前 `confidence` 过于粗糙，只分 `high/low`，而且是结果字段，不是业务判定依据。  
   后续至少需要明确：
   - 哪类命中算 strong match
   - 哪类命中算 fallback match
   - 链接和验证码是否分别判定

4. 明确营销邮件误命中过滤策略  
   这部分不是简单“多加几个黑名单词”，而是要决定：
   - 是否增加邮件级降权
   - 是否增加链接级降权
   - 是否在 fallback 前先做验证语义判断

### 9.2 第二层：建议同步修改的 UI 范围

这部分不是最先止血点，但如果不做，后续仍会长期分叉。

#### A. UI 详情页提取入口

**涉及文件：**

- `static/js/features/emails.js`
- `templates/index.html`

**需要调整的内容：**

1. 详情页按钮不再长期保留前端正则提取  
   当前 `extractVerificationFromDetail()` 在浏览器端直接取正文做正则，这条链路后续应改为调用后端统一接口。

2. 详情页按钮的交互语义需要重新定义  
   例如：
   - 复制高置信度验证码
   - 对低置信度结果给提示
   - 对未命中结果给用户明确反馈

#### B. UI 列表页 / 临时邮箱页提取入口

**涉及文件：**

- `static/js/features/groups.js`
- `static/js/features/temp_emails.js`

**需要调整的内容：**

1. 与新后端返回契约保持一致  
   如果后端新增 strict 语义、低置信度提示或返回结构变化，这两个页面都要同步处理。

2. 统一“复制什么”的行为  
   当前不同入口对复制值的选择可能不同，后续应统一为：
   - 优先复制验证码
   - 无验证码时是否复制验证链接
   - 低置信度时是否只展示不自动复制

### 9.3 第三层：必须补齐的测试范围

如果不补这一层，前两层即使改了，也很容易再次分叉。

**涉及文件：**

- `tests/test_external_api.py`
- `tests/test_verification_extractor.py`
- `tests/test_verification_extractor_options.py`
- `tests/test_api_extract_verification.py`
- 可考虑新增：`tests/test_verification_consistency.py`

**需要补的测试类型：**

1. external API 直接止血测试
   - 营销邮件数字不应被当成高可信验证码
   - 营销链接不应被当成高可信验证链接
   - `confidence=low` 的返回语义要被明确测试

2. 提取器规则测试
   - 标准验证码邮件强匹配
   - fallback 匹配
   - subject/body/html 输入组合差异
   - 候选邮件排序或筛选逻辑

3. 跨链路一致性测试
   - 同一封邮件在 internal API / external API 上结果一致
   - 详情页入口是否仍存在前端独立提取逻辑

### 9.4 第四层：必须同步更新的文档范围

如果实现改了但文档不改，后面会再次出现“代码和口径不一致”。

**涉及文件：**

- `docs/api.md`
- `docs/FD/OPENAPI-00008-对外验证码与邮件读取开放API.yaml`
- `docs/FD/FD-00008-对外验证码与邮件读取开放API.md`
- `docs/TDD/TDD-00008-对外验证码与邮件读取开放API.md`
- `docs/TODO/TODO-00008-对外验证码与邮件读取开放API.md`
- 视需要补充执行日志或发布说明

**需要更新的内容：**

1. `verification-code` / `verification-link` 的成功语义
2. `confidence` 字段的正式定义
3. strict / loose 模式的说明（如果最终采用）
4. UI 与 external API 是否共享统一提取核心的说明
5. 已知限制与接入建议

### 9.5 当前不建议优先修改的范围

这些文件暂时不建议先动，否则容易把问题扩大化：

- `outlook_web/routes/*`
  当前路由边界本身没有成为问题主因，除非后续要新增新接口或新参数。
- 数据库结构 / settings 落库
  当前根因不在持久化层，除非后续引入新的配置开关，例如 strict mode 默认值。
- `wait-message` / `probe` 相关异步链路
  当前问题集中在验证码提取能力，不建议把等待新邮件链路一并拉进首轮整改。

## 十、优先级建议

### 9.1 必须优先处理

1. external API 的成功判定语义  
2. external API 的营销邮件误命中问题

这是直接影响对外调用方的线上风险点。

### 9.2 可以第二阶段处理

1. UI 详情页去掉前端本地正则
2. UI 内部接口与 external API 统一提取核心

2. 更细粒度的严格模式 / 宽松模式能力

## 十三、当前结论

当前验证码提取问题，已经不能再按“调一下正则”来理解。

它的本质是：

- 外部 API 已出现直接业务误判
- UI 与 external API 存在结构性分叉
- 多套提取实现缺少统一契约和统一成功语义

因此，后续应以“先止血，再统一，再补测试”的顺序推进，而不是继续分散处理。
