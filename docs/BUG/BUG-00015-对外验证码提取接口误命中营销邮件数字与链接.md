# BUG-00015 对外验证码提取接口误命中营销邮件数字与链接

> 汇总参考: `docs/BUG/BUG-00017-验证码提取能力准确率与一致性综合问题.md`

> 关联需求: `PRD-00008` / `FD-00008` / `TDD-00008`
> 记录时间: 2026-03-12
> 影响范围: `/api/external/verification-code`、`/api/external/verification-link`
> 当前状态: 已记录，后续评估是否增强提取规则或增加更强过滤条件

---

## 一、问题概述

| 编号 | 标题 | 严重程度 | 类型 |
|------|------|---------|------|
| **V-1** | 验证码/验证链接提取接口在宽松筛选条件下，可能把营销邮件中的普通数字和跳转链接误识别为验证信息 | 🟡 中 | 提取准确率 / 业务语义 |

---

## 二、现象描述

在当前实例联调中，对如下接口发起请求：

```http
GET /api/external/verification-code?email=...&since_minutes=10080
GET /api/external/verification-link?email=...&since_minutes=10080
```

命中的目标邮件实际是一封营销邮件：

- Subject: `Runpod Flash: GPU infrastructure from your terminal`
- From: `team@runpod.io`

但接口仍返回了：

- `verification_code = "1181"`
- `verification_link = "https://..."`（营销邮件中的跳转链接）

从接口响应看，请求是成功的，但从业务语义看，这并不是用户真正想要的“验证码”或“验证链接”。

### 3.1 测试条件

- API Key: 已配置有效 `X-API-Key`

- 查询窗口: `since_minutes=10080`

`/api/external/verification-link` 返回：

```json
  }
}
```

## 四、问题分析

### 4.1 根因判断

当前提取器更偏“通用信息提取”而不是“严格验证码语义识别”：

1. 当邮件正文中出现数字片段时，保底规则可能把普通数字识别为验证码
2. 当邮件正文存在大量链接时，链接提取会返回候选链接，并优先输出其中一个
3. 当调用方不给出更强筛选条件时：
   - 未限定发件人
   - 未限定主题关键词
   - 未限定更窄时间窗口
   - 未限定 `code_regex` / `code_length`
   接口就可能从“最近一封普通邮件”中提取出并不符合业务预期的结果

### 4.2 深层根因拆解

这次误命中的直接机制，不只是“正则太宽松”，而是一条完整链路共同造成：

1. 邮件选择策略偏宽松  
   当前 `/api/external/verification-code` 与 `/api/external/verification-link` 的实现，不是“在候选邮件里寻找最像验证码邮件的一封”，而是：
   - 先按 `from_contains` / `subject_contains` / `since_minutes` 做基础过滤
   - 直接取符合条件的最新一封邮件
   - 再对这封邮件做验证码/链接提取

   这意味着只要筛选条件较宽，最新一封恰好是营销邮件，就会把后续提取暴露在错误样本上。

2. 验证码 fallback 规则偏“尽量提取”  
   当关键词强匹配失败后，当前实现会全局扫描正文中的 `4-8` 位数字候选，并返回第一个符合基本过滤规则的值。

   这套逻辑适合“尽可能帮用户找到一个候选值”，但不适合“直接作为自动化验证码输入”。

3. 链接 fallback 规则偏“尽量返回一个链接”  
   当前实现会优先挑选包含 `verify` / `confirm` / `activate` / `validation` 等关键词的链接；如果没有命中，则直接回退到正文中的第一条有效外链。

   对营销邮件来说，这几乎等价于“总能提一个链接出来”，但这个链接未必具备验证语义。

4. `confidence=low` 未参与接口成功判定  
   当前接口虽然已经返回 `confidence` 字段，但 controller 层只要发现：
   - `verification_code` 非空
   - 或 `verification_link` 非空

   就直接按成功返回 `200 OK`。

   换句话说，系统已经知道这是“低置信度候选”，但还没有把它当作失败或待确认结果处理，这才让误判真正落到了调用方侧。

### 4.3 当前接口行为是否错误

从“程序是否按规则运行”角度看，不属于接口不可用：

- 接口返回结构正确
- 没有报错
- 提取逻辑确实找到了数字和链接

但从“业务结果是否可信”角度看，这属于明显的准确率缺陷：

- 误把营销内容当成验证信息
- 容易让调用方误自动填错验证码或跳到错误链接

### 4.4 根因归类

本问题建议归类为“三段式根因”：

1. 直接原因  
   外部 API 当前采用“最新邮件 + 宽松 fallback + low confidence 仍成功”的组合策略。

2. 设计原因  
   当前提取器更偏“候选信息提取器”，而不是“严格验证码识别器”，但接口语义却被调用方理解为“可直接自动化消费的验证码接口”。

3. 结构原因  
   系统内部存在多套提取逻辑并存，缺少统一的提取核心与统一的成功判定标准。这也是 `BUG-00016` 要单独记录的原因。

## 五、影响评估

### 5.1 对自动化接入的影响

- 自动注册/自动验证流程可能误用无关数字
- 回调或自动点击场景可能跳到营销链接而不是验证链接
- 调用方会误认为“接口成功但业务失败”，排查成本高

### 5.2 对当前版本的风险级别

- 对“宽松条件下直接自动化消费结果”的调用方有实际风险
- 对“带明确筛选条件 + 人工复核”的调用场景风险较低

---

## 六、当前处理口径

当前阶段先按以下方式处理：

1. 记录为提取准确率问题，而不是接口稳定性问题
2. 建议调用方在生产使用时增加筛选条件：
   - `subject_contains`
   - `from_contains`
   - 更小的 `since_minutes`
   - 需要时指定 `code_length` 或 `code_regex`
3. 对 `confidence=low` 的结果，不建议直接作为强自动化输入

---

## 七、后续可选改进

可选方向：

1. 不再默认“最新一封即目标邮件”，而是在候选邮件中优先寻找更像验证码邮件的样本
2. 增强验证码提取规则，降低普通数字误判概率
3. 增强验证链接优先级规则，降低营销跳转链接误判概率
4. 对明显营销邮件特征增加降权或过滤
5. 在接口层让 `confidence` 参与成功判定，而不是只做展示字段
6. 为调用方提供“仅强匹配模式”或更严格的提取选项

当前不建议仓促把所有普通数字都过滤掉，否则容易把真实验证码邮件也一起误杀。
