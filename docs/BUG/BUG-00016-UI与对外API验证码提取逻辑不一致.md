# BUG-00016 UI 与对外 API 验证码提取逻辑不一致

> 汇总参考: `docs/BUG/BUG-00017-验证码提取能力准确率与一致性综合问题.md`

> 关联需求: `PRD-00008` / `FD-00008` / `TDD-00008`
> 记录时间: 2026-03-12
> 影响范围: UI 列表页提取、UI 详情页提取、`/api/external/verification-code`、`/api/external/verification-link`
> 当前状态: 已记录，建议后续按“统一提取核心”方向协调整改

---

## 一、问题概述

| 编号 | 标题 | 严重程度 | 类型 |
|------|------|---------|------|
| **C-1** | 当前系统内“提取验证码/链接”并非一套统一实现，而是三套逻辑并存，导致 UI 与对外 API 结果可能不一致 | 🟠 高 | 业务一致性 / 架构协同 |

---

## 二、结论摘要

当前“提取验证码”能力至少存在三条不同链路：

1. UI 列表页/分组页按钮
2. UI 邮件详情页按钮
3. 对外开放 API

这三条链路不是同一个提取器，也不是同一套规则，因此不能认为“UI 的提取逻辑”和“对外 API 的提取逻辑”是一致的。

`BUG-00015` 主要暴露的是“对外 API 参数化提取链路”的准确率问题，但不能简单等同于 UI 详情页按钮的提取问题。

---

## 三、现状拆解

### 3.1 UI 列表页 / 分组页

列表页“提取验证码”会请求内部接口：

```text
GET /api/emails/{email}/extract-verification
```

该接口最终调用的是旧提取链路：

```python
extract_verification_info(email_obj)
```

其特点是：

- 走后端内部接口
- 使用旧版提取器
- 不带外部 API 那些参数化能力

### 3.2 UI 邮件详情页

详情页按钮不是调用后端统一接口，而是在前端直接对当前正文做本地正则提取。

其特点是：

- 直接从 DOM / iframe 读取正文文本
- 前端用正则抓 `4-8` 位数字或字母
- 同时从正文里抓第一个看起来像链接的 URL

这意味着详情页按钮本质上绕过了后端提取服务。

### 3.3 对外开放 API

对外接口：

```text
GET /api/external/verification-code
GET /api/external/verification-link
```

最终调用的是扩展提取链路：

```python
extract_verification_info_with_options(...)
```

其特点是：

- 支持 `code_regex`
- 支持 `code_length`
- 支持 `code_source`
- 支持链接关键词优先级
- 带 `confidence` 概念

这套逻辑明显比 UI 内部旧提取器和详情页前端正则更复杂。

---

## 四、问题表现

由于三条链路提取规则不同，当前会出现以下现象：

1. 同一封邮件，在 UI 详情页点“验证码”得到的结果，可能和列表页按钮不一样
2. 同一封邮件，在 UI 提取到验证码，但外部 API 提取不到，反之也可能成立
3. 外部 API 命中了营销邮件中的普通数字或普通跳转链接，但 UI 某个入口未必会复现完全相同结果
4. 用户和接入方会误以为“提取能力”是同一套语义，实际却是不同实现

---

## 五、根因分析

### 5.1 缺少统一提取核心

系统没有把“验证码/验证链接提取”收敛成一个单一后端核心服务，而是随着功能扩展逐步分叉为：

- 旧版内部提取器
- 外部 API 参数化提取器
- 前端本地正则提取

### 5.2 UI 为了交互便利保留了前端本地提取

详情页按钮直接在浏览器端做正则，短期上实现快，但长期会带来：

- 行为难以和服务端保持一致
- 修规则时必须多处同步
- 测试覆盖也难以统一

### 5.3 外部 API 需求扩展后与内部能力分叉

为了支持更灵活的开放 API，新增了带参数的提取器，但内部 UI 仍停留在旧逻辑，最终形成双轨甚至三轨。

### 5.4 更深层的根因不是“实现失误”，而是契约缺失

当前系统里“提取验证码”这个能力名是统一的，但底层实际上没有统一契约。缺失的关键契约包括：

1. 输入契约  
   到底应该基于哪些内容提取：
   - 只看正文文本
   - 还是正文 + HTML
   - 还是正文 + HTML + subject
   - 是否允许回退到 preview

2. 规则契约  
   到底什么叫“验证码”：
   - 关键词附近命中的数字
   - 任意 `4-8` 位数字
   - 含字母数字组合
   - 是否允许全局 fallback

3. 成功契约  
   到底什么情况下接口应该返回成功：
   - 只要提取到任意候选值就成功
   - 还是只有高置信度结果才成功
   - 低置信度结果是返回警告还是返回失败

4. 一致性契约  
   UI、内部 API、对外 API 是否必须在同一封邮件上给出一致结果。

因为这些契约没有被明确下来，所以不同入口都用“最适合当前场景”的方式各自演进，最终形成结构性分叉。

### 5.5 输入模型不统一，是结果不一致的直接技术原因

三条链路不仅规则不同，连输入源都不同：

1. 旧提取器  
   通常基于 `body -> body_html转文本 -> body_preview -> subject` 的顺序回退。

2. 外部 API 提取器  
   默认会把 `subject + content + html` 拼接后再做参数化提取。

3. UI 详情页前端提取  
   直接读取浏览器中渲染后的正文文本，再用前端正则抓值和抓链接。

这意味着：

- 有的链路会把 `subject` 算进候选范围
- 有的链路会把原始 HTML 文本算进去
- 有的链路只看浏览器可见文本
- 有的链路取“第一条链接”
- 有的链路按关键词优先

输入模型不统一，结果一致只能靠运气，不可能长期稳定。

### 5.6 测试没有约束“跨链路一致性”

当前测试更多是在验证：

- 旧提取器可用
- 新提取器可用
- 对外 API 可用
- UI JS 函数存在

但没有把下面这些要求纳入自动化约束：

1. 同一封标准验证码邮件，在 UI / 内部 API / 对外 API 上是否结果一致
2. 同一封营销邮件，在三个入口上是否都不会被当成可信验证码
3. `confidence=low` 是否应影响最终接口返回语义

这使得多套实现即使长期分叉，也不会在测试阶段被快速暴露。

## 六、影响评估

### 6.1 对产品语义的影响

用户看到的按钮名称都是“提取验证码”，自然会认为结果应一致。当前实现会破坏这一预期，属于产品语义不一致问题。

### 6.2 对维护成本的影响

后续每次修复提取规则时，都需要分别判断：

- 改前端详情页正则
- 改内部 API 旧提取器
- 改外部 API 扩展提取器

这会持续放大维护成本和回归风险。

### 6.3 对测试体系的影响

即使单元测试通过，也不能证明三条用户可见链路的行为一致。当前测试更容易验证“各自可用”，但难以验证“整体一致”。

---

## 七、与其他 BUG 的关系

### 7.1 与 `BUG-00014` 的关系

`BUG-00014` 是对外消息详情接口的接入兼容性问题，和提取逻辑一致性不是同一层问题。

### 7.2 与 `BUG-00015` 的关系

`BUG-00015` 是外部 API 提取准确率问题。

本问题则更上层，属于“同一业务能力存在多套实现”的一致性问题。两者关系如下：

1. `BUG-00015` 是这次暴露出来的具体症状之一
2. 本问题是导致后续类似问题反复出现的结构性原因

---

## 八、当前处理建议

当前建议先按“架构协调类 BUG”处理，而不是继续零散修补单个入口：

1. 明确唯一可信的提取核心应放在后端服务层
2. UI 详情页不再长期保留前端独立正则提取
3. UI 内部接口与对外 API 最终应收敛到同一个提取核心
4. 差异应通过参数体现，而不是通过不同实现体现

补充建议：

5. 在统一提取核心前，不应默认宣称“UI 提取结果”和“external API 提取结果”语义一致
6. 后续修复 `BUG-00015` 时，应优先在统一契约下修，不要继续单点堆补丁

---

## 九、建议整改顺序

建议按最小风险顺序推进：

1. 先把 UI 详情页按钮改为调用后端统一接口，去掉浏览器端独立正则逻辑
2. 再协调内部 UI 接口与对外 API 的提取核心，统一到同一个 service
3. 最后补齐一致性测试，覆盖：
   - 同一封邮件在 UI/外部 API 的结果对比
   - 营销邮件误命中场景
   - 验证码邮件强匹配场景

---

## 十、当前结论

这个问题有必要单独立项处理。

原因不是“代码风格不统一”这么简单，而是：

- 已经影响业务语义一致性
- 已经影响 BUG 判断边界
- 已经开始增加维护和测试成本
- 已经让 `BUG-00015` 这类准确率问题难以被单点修复

因此，后续不建议继续把 UI 和外部 API 当成互相默认一致的能力来讨论，应先完成提取核心协调，再继续修准确率和交互细节。
